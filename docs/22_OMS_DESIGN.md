# 22 — Order Management System (OMS)

**Platform:** NSE · Indian Equity FnO  
**Position in Pipeline:** Risk Engine → OMS → Broker → Exchange  
**Version:** 1.0  

---

## Design Principles

- The OMS is the **sole** component that submits orders to the broker. No other service calls `broker.place_order()`.
- Every order state change is an immutable event appended to `order_events`. Nothing is updated in-place.
- The OMS is idempotent. Receiving the same `signal.risk.approved` event twice produces only one order.
- Kill switch is checked synchronously before every **new** order placement. Cancellations bypass this check (see Doc 14).
- The OMS does not generate signals, compute risk, or make trading decisions. It executes what the Risk Engine approves.
- Paper mode is a first-class OMS mode — it mirrors all logic but never calls the real broker.

---

## Architectural Position

```
[signal.risk.approved event] ──► OMS ──► IBroker.place_order() ──► Exchange
                                   │
                                   ├── Kill switch check (synchronous, in-memory)
                                   ├── Idempotency check (Redis dedup key)
                                   ├── Signal TTL check (reject if expired)
                                   ├── Rate limit check (max orders/minute)
                                   ├── Order state machine (PENDING → OPEN → ...)
                                   └── order_events append (every state transition)
```

## Order State Machine

```
PENDING
  │ OMS receives signal.risk.approved, passes all pre-submission checks
  ↓
SUBMITTING
  │ OMS calls broker.place_order()
  ├──→ REJECTED_PRE_SUBMIT   — kill switch, TTL expired, rate limit, idempotency block (terminal)
  └──→ SUBMITTED             — broker API call succeeded (order_id received from broker)
         │
         │ broker callback: order acknowledgement
         ↓
       OPEN                  — exchange accepted the order, sitting in order book
         │
         ├──→ FILLED           — fully executed at exchange (terminal → creates POSITION)
         ├──→ PARTIALLY_FILLED — partial execution; remainder stays OPEN
         ├──→ CANCELLED        — cancelled by OMS, operator, or kill switch (terminal)
         ├──→ REJECTED         — exchange or broker rejected (terminal, reason logged)
         └──→ EXPIRED          — order not filled before market close (terminal)

PARTIALLY_FILLED:
  ├──→ FILLED     — remaining quantity filled
  └──→ CANCELLED  — remaining quantity cancelled
```

Illegal transitions raise `OrderStateError` and are logged at ERROR level.  
Every transition is appended to `order_events` (hypertable, append-only).

## Pre-Submission Checks

Executed synchronously in this order before calling `broker.place_order()`:

| Check | Condition | Action on Failure |
|-------|-----------|------------------|
| Kill switch | `oms.kill_switch_flag == True` | Raise `KillSwitchActiveError`; publish `order.rejected` |
| Signal TTL | `current_time >= signal.valid_until` | Raise `SignalExpiredError`; publish `order.rejected` |
| Idempotency | Redis key `oms:idem:{signal_id}` exists | Log and silently discard (not an error) |
| Order rate limit | Orders in last 60s >= `max_orders_per_minute` | Raise `OrderRateLimitError`; publish `order.rejected` |
| Trading mode | `trading_mode == PAPER` | Route to PaperBrokerAdapter instead of live broker |
| Instrument active | `InstrumentMaster.is_active(instrument)` | Raise `InstrumentNotActiveError` |

Idempotency key TTL = 5 minutes. Format: `oms:idem:{signal_id}`.  
Set atomically with `SET NX PX 300000` before calling the broker.

## Order Type Selection

The OMS selects order type based on signal context and configuration:

```
For FnO intraday signals (default):
  Primary order:   MARKET (price priority — avoid slippage risk from limit miss)
  Stop-loss order: SL_MARKET (guaranteed fill when stop is hit)
  
  Override to LIMIT when:
    Option premium > 500 (large premium; slippage on MARKET is significant)
    OR config: order_type_override = LIMIT

For LIMIT orders:
  entry_limit_price = signal.entry_zone_high × 1.001  (0.1% above for buys)
                    = signal.entry_zone_low  × 0.999  (0.1% below for sells)
  IOC validity (Immediate or Cancel) — prevents stale limit orders from filling in changed conditions
```

## Stop-Loss Order Management

After the primary order fills, the OMS immediately places the stop-loss order:

```
Stop-loss placement sequence (on FILLED event):
  1. Receive order.filled event
  2. Compute stop_price from signal.stop_loss (pre-computed by PositionSizer)
  3. Place SL_MARKET order with trigger_price = stop_price
  4. Publish order.stoploss.placed event
  5. Store stop order in oms:stop:{position_id} Redis key

On stop-loss fill:
  1. Receive order.filled event for stop order
  2. Publish position.closed with outcome=STOP_HIT
  3. Write to signal_performance_stats
  4. Clear Redis keys for this position
```

**Stop-loss must be placed within 2 seconds of primary fill.** A monitoring task alerts if this deadline is missed.

---

## Target Order Management

After primary fill, the OMS places a limit order at the primary target:

```
Target placement sequence:
  1. Place LIMIT order at signal.target_1 (T1)
  2. Store as oms:target:{position_id} in Redis

Partial target strategy:
  T1 hit → close 50% of position at T1, move stop to breakeven, hold rest for T2
  T2 hit → close remaining 50% at T2
  
  This is configurable per instrument class:
    config: target_strategy = SINGLE | PARTIAL_SCALE
```

## Position Reconciliation

After every WebSocket reconnection, the OMS runs a reconciliation pass:

```
ReconciliationService.reconcile():
  1. Fetch all orders with state OPEN or SUBMITTING from the DB
  2. Call broker.get_orders() for the current session
  3. For each discrepancy:
     a. Order in DB as OPEN but not at broker → check if filled or cancelled broker-side
     b. Order at broker not in DB → log CRITICAL (unexpected order — possible rogue order)
     c. Fill price differs by > 0.5% → log WARNING (slippage alert)
  4. Apply state corrections
  5. Publish position.reconciled event
```

An unexpected order at the broker (not originated by this OMS) triggers an **immediate kill switch activation** with reason `ROGUE_ORDER_DETECTED`.

## Paper Trading Mode

When `trading_mode == PAPER` (set in Redis by Risk Engine graduated response):

```
PaperBrokerAdapter behavior:
  place_order()      → simulate fill at current LTP from Redis; no broker API call
  cancel_order()     → mark as cancelled; no broker API call
  get_orders()       → return from OMS internal DB, not broker
  get_positions()    → return from OMS internal DB
  get_margins()      → return configured paper capital; no broker API call

Fill simulation:
  MARKET order: fills at LTP ± 0.05% (simulated slippage, configurable)
  LIMIT order:  fills when LTP crosses limit price (checked every tick)
  SL_MARKET:    fills at LTP when trigger is breached (slippage = 0.10%)
```

Paper mode is transparent to all downstream consumers — position, P&L, and analytics services behave identically. Only `trading_mode = PAPER` in the audit trail distinguishes it.

## Order Events Schema

Every order state transition is appended to `order_events` (hypertable, see Doc 18).

Key event types:

| `event_type` | Trigger | `event_data` fields |
|-------------|---------|---------------------|
| `ORDER_CREATED` | OMS creates order object | full order schema |
| `ORDER_SUBMITTED` | broker.place_order() called | broker_order_id, submitted_at |
| `ORDER_FILLED` | broker callback / webhook | fill_price, fill_qty, fill_time |
| `ORDER_PARTIAL_FILL` | partial execution | filled_qty, remaining_qty, avg_price |
| `ORDER_CANCELLED` | cancel call completed | cancel_reason, cancelled_by |
| `ORDER_REJECTED` | broker/exchange rejected | rejection_reason, broker_message |
| `ORDER_EXPIRED` | market close with no fill | — |
| `KILL_SWITCH_BLOCKED` | kill switch prevented submission | kill_switch_activated_at |
| `TTL_EXPIRED` | signal expired before submission | signal_valid_until, current_time |
| `IDEMPOTENCY_BLOCK` | duplicate signal_id detected | original_order_id |
| `STOPLOSS_PLACED` | stop order placed after fill | stop_order_id, trigger_price |
| `TARGET_PLACED` | target order placed after fill | target_order_id, limit_price |
| `PAPER_SIMULATED_FILL` | paper mode fill | simulated_price, ltp_at_fill |

## Latency Budget

| Operation | SLO | Notes |
|-----------|-----|-------|
| signal.risk.approved → order submitted to broker | < 500ms P99 | Includes all pre-submission checks |
| Broker order → OMS acknowledgement | < 2s P99 | Broker API budget |
| Fill event → stop-loss order placed | < 2s P99 | Safety-critical |
| Kill switch check within place_order() | < 1ms | In-memory flag read |
